# Result Explainability View

**Date:** 2026-03-12
**Goal:** Let an organizer (or parent, post-publish) tap any dancer's result and see exactly how it was calculated — per-judge scores, ranks, points, and rules applied.
**Truth test questions addressed:** #15 (explain results)

---

## What We're Building

Expandable rows in the existing `ResultsTable` component. Tap a dancer to see a frozen snapshot of how their result was computed. Same component serves both the dashboard (organizer) and public results page (after publish).

This is a **published snapshot**, not a live recalculation. The breakdown explains the saved result at the moment tabulation ran. If scores are later corrected and re-tabulated, the snapshot updates with the new tabulation. The data is self-contained — no live queries needed to render it.

---

## Architecture: Engine Stays Pure

The tabulation engine (`src/lib/engine/`) is not modified. It continues to return `TabulationResult` with `individual_ranks`. The enrichment happens in the **caller** (competition detail page) when building `calculated_payload` for storage.

**Why:** The engine is pure math — no display concerns, no judge names, no flag metadata. Threading display data through the engine violates CLAUDE.md section 1.4. The caller already has access to judges, scores, and registrations. It builds the richer payload by cross-referencing engine output with its own data.

---

## Data Model: Extended `calculated_payload`

The `calculated_payload` JSONB in the `results` table currently stores:

```ts
{
  total_points: number
  individual_ranks: { judge_id: string; rank: number; irish_points: number }[]
  rules_snapshot: RuleSetConfig
}
```

**Change:** Add `judge_scores` alongside `individual_ranks` (old field kept for backward compat):

```ts
{
  total_points: number
  individual_ranks: { judge_id: string; rank: number; irish_points: number }[]  // kept for compat
  judge_scores: {
    judge_id: string
    judge_name: string        // frozen at tabulation time
    raw_score: number
    rank: number
    irish_points: number
    flagged?: boolean         // only present when true
    flag_reason?: string      // only present when flagged
  }[]
  rules_snapshot: RuleSetConfig
  tie_break_applied: boolean
  tie_break_note: string | null   // e.g., "Won countback: 2 firsts vs 1 first"
  drop_applied: false             // placeholder — drop rules not yet implemented
  drop_note: null                 // placeholder — always null until drop rules ship
}
```

**Why freeze judge names:** Published results may be viewed long after the event. Resolving judge UUIDs at render time requires a live query and could fail if judges are later removed. Freezing the name makes the explanation fully self-contained.

**Why `flagged`/`flag_reason` are optional:** Most scores aren't flagged. Omitting these fields on unflagged scores keeps the payload clean.

**Migration:** No schema migration needed — `calculated_payload` is JSONB, so the shape change is handled in application code only.

## Building `judge_scores` in the Caller

The competition detail page's `handleTabulate` function already has:
- `scores[]` — all `score_entries` rows (with `raw_score`, `flagged`, `flag_reason`, `judge_id`, `dancer_id`)
- `judges[]` — all judges (with `id`, `first_name`, `last_name`)
- `tabulationResults[]` — engine output (with `individual_ranks`)

After calling `tabulate()`, the caller builds `judge_scores` for each result:

```ts
// For each TabulationResult:
const judgeScores = result.individual_ranks.map(ir => {
  const judge = judges.find(j => j.id === ir.judge_id)
  const score = scores.find(s =>
    s.dancer_id === result.dancer_id && s.judge_id === ir.judge_id
  )
  return {
    judge_id: ir.judge_id,
    judge_name: judge ? `${judge.first_name} ${judge.last_name}` : ir.judge_id,
    raw_score: score ? Number(score.raw_score) : 0,
    rank: ir.rank,
    irish_points: ir.irish_points,
    ...(score?.flagged ? { flagged: true, flag_reason: score.flag_reason ?? undefined } : {}),
  }
})
```

This enrichment helper should be a pure function in a new file `src/lib/result-payload.ts` — not in the engine, not inline in the page. It takes engine output + judges + scores and returns the enriched payload. Testable independently.

## Tie-Break Detection

The caller detects whether tie-break was applied by inspecting the engine results:

- If two results have the same `total_points` but different `final_rank` → countback was applied, one won
- If two results have the same `total_points` and same `final_rank` → true tie (countback didn't resolve)

The `tie_break_note` is generated by comparing the `individual_ranks` of tied dancers using the same countback logic the engine uses — counting 1st places, then 2nd places, etc. This note generation belongs in `src/lib/result-payload.ts`.

**When tie-break is not applicable:** `tie_break_applied: false`, `tie_break_note: null`. The UI doesn't show the banner.

---

## Drop Rules: Scoped Out

Drop rules (`drop_high`, `drop_low`) exist in `RuleSetConfig` but are **not implemented in the tabulation engine**. This spec does not implement them.

`drop_applied` is always `false` and `drop_note` is always `null`. The fields exist in the payload type so they're ready when drop rules are implemented later. The UI handles them (dimmed rows, banner) but won't encounter them until the engine supports drops.

## UI: Expanded Row Panel

### Layout

When a dancer's row is tapped, it expands to show a breakdown panel directly below the row. The row itself stays visible. Tapping again collapses it. Only one row can be expanded at a time.

### Panel Content

**Top summary line (anchors the panel immediately):**
> **Total Irish Points: 215 · Final Place: 3rd**

**Callout banners (only when applicable, not buried in footer):**
- If tie-break applied: `"Tie-break applied — Won countback: 2 firsts vs 1 first"`
- If drop rules applied (future): `"Highest and lowest judge scores removed"`
- Styled as a subtle info banner, not alarming. Only present when these rules were used.

**Per-judge table:**

| Judge | Raw Score | Rank | Irish Points |
|-------|-----------|------|-------------|
| Mary O'Brien | 72.5 | 3rd | 65 |
| Patrick Kelly | 78.0 | 2nd | 75 |
| Bobby Filet | 76.0 | 2nd | 75 |
| **Total** | | | **215** |

- Flagged scores: small warning icon next to the judge name, flag reason shown as muted text below. Don't make the entire panel look like a crisis — a small indicator and reason is enough.
- Dropped scores (future, when drop rules ship): dimmed row with strikethrough on the Irish Points value.

**Rules footer (human-readable, never raw config):**
> Irish Points scoring (1st=100, 2nd=75, 3rd=65...) · Score range: 0–100

Generated in the UI component by reading `rules_snapshot`. This is presentation logic, not engine logic.

### Styling

- Panel background: `bg-feis-green-light` or similar light tint to distinguish from table rows
- Inner card: white with border, matching `feis-card` pattern
- Monospace for scores and points (precision utility aesthetic)
- Compact — should not require scrolling for 3 judges

## Backward Compatibility

Existing results in the database have `individual_ranks` (old format) but no `judge_scores`. The `ResultsTable` component must handle both:

- If `judge_scores` exists → render full breakdown with raw scores
- If only `individual_ranks` exists → render partial breakdown (rank + points per judge, no raw score, judge_id instead of name)
- If neither → don't show expand arrow

This avoids requiring a data migration. Re-tabulating a competition produces the new format. Old results degrade gracefully.

---

## What This Does NOT Include

- Comparison between dancers (no "why did I lose to her?")
- Entry mode or audit trail (that's on the audit page)
- Score editing from the explainability view
- Real-time updates (page refresh to see changes)
- Shareable permalink per dancer (could add later, not needed for pilot)
- Drop rule implementation (fields are placeholder)

## Acceptance Criteria

1. Tapping a dancer row in results expands to show per-judge breakdown
2. Breakdown shows: judge name, raw score, rank, Irish Points per judge
3. Top summary line shows total points and final place
4. Tie-break banner shown prominently when countback was applied
5. Flagged scores shown with small warning indicator and reason, not alarmist
6. Rules footer is human-readable, never raw config
7. Data is a frozen snapshot from tabulation time, not a live query
8. Judge names are frozen in the payload, not resolved at render time
9. Works on both dashboard results and public results page
10. Backward compatible with old `individual_ranks` format (graceful degradation)
11. Only one row expanded at a time
12. No other dancers' scores shown
13. Tabulation engine is not modified — enrichment happens in caller
14. Drop rule fields are placeholder (always false/null)
